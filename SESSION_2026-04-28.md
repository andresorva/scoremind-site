## 5. i18n SYSTEM

`/assets/js/i18n.js` is a single IIFE that:

1. **Auto-detects initial language** in this priority order:
   - URL param `?lang=en` or `?lang=es` (used for forced-language testing/screenshots)
   - `localStorage.getItem('sm_lang')`
   - `navigator.language.startsWith('es')` → `'es'`, else `'en'`

2. **Applies translations** by:
   - Setting `<html lang="...">`
   - Iterating `[data-i18n="key"]` and setting `textContent` from the dictionary
   - Iterating `[data-i18n-attr="attr1:key1, attr2:key2"]` and setting attributes (used for image alt text, etc.)
   - Updating `<title>` and `<meta name="description">` from `meta.title` / `meta.description` keys
   - Toggling the `.active` class on `[data-lang-btn]` buttons + setting `aria-pressed`

3. **Wires click handlers** on each `[data-lang-btn]` to call `setLang(lang)`, which writes to localStorage and re-applies.

4. **Exposes a global** `window.SMI18n.setLang(lang)` for programmatic switching (and `getDict()` for debugging).

**Dictionary structure:** flat keys with dot notation, `{ en: {...}, es: {...} }`. About 70 keys total.

**Key naming convention:** `<section>.<element>` or `<section>.<element>_<modifier>`. E.g., `hero.btn_primary_small`, `trust.card3.label`. Stay consistent if you add keys.

## 6. CUSTOM QR CODE GENERATION

Generated **once at design time**, committed as static SVGs. Zero runtime JS overhead.

**Generator script** (one-shot, lives in `/tmp/scoremind-qr-gen/`, NOT in repo):
- Uses `qrcode` npm package for the QR matrix
- Custom SVG renderer:
  - Modules → `<circle>` with `r = cellSize * 0.42` (rounded look)
  - Skips finder patterns (the 3 corner squares)
  - Renders finder patterns as: outer rounded square (white), inner rounded square (black), center dot circle (cyan)
- Error correction level **H** (30% recovery — necessary because we overlay the logo on top)
- 8px cell size, 4-cell margin = nice padding

**The 2 generated SVGs:**
- `assets/qr/google-play.svg` — points to `https://play.google.com/store/apps/details?id=ai.scoremind.app`. Cyan modules (#00CEC9) on black. White rounded eyes with cyan inner dots. 45×45 modules (44 alphanumerics + EC).
- `assets/qr/app-store-dimmed.svg` — points to `https://scoremind.ai` as placeholder. Grayscale modules (#666666) on black. Visually muted to signal "not yet active". 29×29 modules (the URL is shorter so fewer cells).

**Logo overlay:** the QRs do NOT have the logo embedded inside the SVG. Instead, in `index.html` the QR is wrapped with the logo positioned absolutely on top via CSS (`.qr-logo`). This keeps the SVG simple and lets us reuse the same logo file site-wide.

**To regenerate** (e.g., when Apple approves and we have the real App Store URL): edit `/tmp/scoremind-qr-gen/gen.js`, run `node gen.js`. It writes directly into `/Users/mac/scoremind-site/assets/qr/`. The generator script content is documented in this session's chat history if needed.

## 8. BUGS ENCOUNTERED & RESOLVED DURING SESSION

These are documented because they may resurface or inform future debugging:

### Bug A — Hero with 3 phones got cut off at viewports <1440px
- **Symptom:** Side phones (`match.webp`, `scorebot.webp`) showed only partial text — "Bey..." instead of "Beyond" — at Mac's 1290px laptop viewport.
- **Diagnosis:** At 1290px viewport, the hero right column is 589px wide. Center phone is 280px (140px half). For a side phone scaled 0.85 (238px wide, 119px half) to NOT overlap the center phone, translateX must be ≥ 259px. But to fit in the 589px column, translateX ≤ 175.5px. **Mathematically impossible** to show 3 phones non-overlapping at scale 0.85+ in a 589px column.
- **Resolution:** Switched to single-phone hero across ALL viewports (Mac's call: "1 phone is better than cut phones"). The other 4 screenshots got their own dedicated sections (Meet ScoreBot, Beyond the Score, Stats That Matter, Live Social Proof card). This matches Linear/Raycast/Vercel pattern.

### Bug B — Hero phone invisible in Chrome headless screenshots
- **Symptom:** Headless screenshots showed huge empty space where the hero phone should be.
- **Diagnosis:** Original CSS had `opacity: 0` as the default state and an animation set the visible state. With `--virtual-time-budget`, the animation may complete but the screenshot got captured mid-animation.
- **Resolution:** Default state of `.hero-phone` is now visible (`opacity: 1; transform: rotate(2deg)`). The animation has `from { opacity: 0; transform: ... translateY(20px); } to { ... }` so it still animates IN on first paint, but if the animation never fires, the phone is still visible.

### Bug C — Mobile horizontal overflow (484px scrollWidth at 375px viewport)
- **Symptom:** Mobile full-page screenshots came out 484px wide (109px overflow). Body had `overflow-x: hidden` so users wouldn't see it, but it was there.
- **Diagnosis:** Run a `getBoundingClientRect()` audit at 375vp via puppeteer revealed the offenders: `.trust-card` elements at `left: 279, right: 484, width: 205`. The 2-column trust grid was forcing each card to fit a `12,000+` number at 48px font weight 800 = ~155px min-content, which (plus padding) made cards 205px wide. With 2-col grid and gap, that's 426px > 335px container.
- **Resolution:** `grid-template-columns: repeat(2, minmax(0, 1fr))` at ≤1023px allows columns to shrink below content min-width. At ≤767px, drop to 1 column entirely + reduce trust-number font to 40px. Final scrollWidth = 375px exactly.

### Bug D — Mobile h1 truncation false alarm
- **Symptom:** Two screenshots in a row showed "The Future of Football Predict" with the rest cut off.
- **Diagnosis:** Stale screenshot cache (the screenshot path was reused but Chrome user-data-dir was caching). After switching to puppeteer with proper `await page.goto`, the issue disappeared. Verified via `getComputedStyle`: h1 is 40px, offsetWidth 335px, fits in 335px container.
- **Lesson:** Always validate visual claims with computed styles before changing code.

### Bug E — Sub-agents blocked by sandbox permissions
- **Symptom:** Spawned 3 sub-agents in parallel (icons, PNG backup, CLAUDE.md updates). All 3 returned "permission denied" on their first Write/Edit/Bash call.
- **Resolution:** Did the work directly in the main session. Sub-agents may have a more restricted permission model than the main thread. **For future sessions:** if you spawn agents, give them tasks that fit their tool set, but always have a fallback plan to do the work in main thread if they get permission errors.
